## 2. 这套代码的底层逻辑

这套代码不是“拿历史 JSONL 去训练一个模型”。

它的核心逻辑是：

1. 先把模型导成推理计算图。
2. 对图上的节点做解析式估算，得到每个 node 的 `flops / bytes / estimated_time_ms`。
3. 如果有模块级 profiling，就把某些节点组合成模块范围，例如 `self_attn / mlp / layernorm`。
4. 用模块 profiling 的实测值去覆盖这些节点的解析估算值。
5. 对 TP 场景再额外加通信预测，把各 phase 组合回 `prefill / decode_step / request`。

也就是说：

- `JSONL` 是一个“模块 profile 查表库”。
- 不是机器学习训练集。
- 不是参数拟合出来的黑盒模型。
- 更接近“解析模型 + 查表修正 + 通信启发式组合”。

对应代码位置：

- 图导出与运行输入准备：[mvp_runtime.py](/home/o_mabin/hanwuji-clj-proj/projects/inference/time-modeling/mvp_runtime.py)
- 节点估算与通信预测：[mvp_estimator.py](/home/o_mabin/hanwuji-clj-proj/projects/inference/time-modeling/mvp_estimator.py)
- 在线模块 profiling：[mvp_profile.py](/home/o_mabin/hanwuji-clj-proj/projects/inference/time-modeling/mvp_profile.py)
- JSONL 查表/写表：[mvp_table.py](/home/o_mabin/hanwuji-clj-proj/projects/inference/time-modeling/mvp_table.py)
- 主流程编排：[mvp_app.py](/home/o_mabin/hanwuji-clj-proj/projects/inference/time-modeling/mvp_app.py)

## 3. JSONL Profiling 表在当前实现中的角色

当前实现支持三种模式：

- `online`
  - 每次运行都现场采集模块级 profiling。
- `table`
  - 只从历史 JSONL 里查，不现场采。
- `hybrid`
  - 先查 JSONL，缺的 scope 再现场补采。

对应代码：

- `mvp_app.py` 中 `estimate_mode` 分支
- `mvp_table.py` 中 `load_module_profiles_from_table()` 和 `append_module_profiles_to_table()`

JSONL 表的 key 不是随便复用的，它至少包含：

- `model_id`
- `dtype`
- `prompt_tokens`
- `parallel_mode`
- `tp_size`
- `world_size`
- `nnodes`
- `interconnect`
- `device_name`
- `phase`
- `module_scope`

因此：

- 英伟达上录的 `NVIDIA A100` profile 不适合直接给 `MLU580` 用。
- 寒武纪上应该单独录一份表。
- 本轮已经开始这样做，并单独落到了 `module_profile_table_cambricon_mlu580.jsonl`。

## 8. 下一步建议

优先级建议如下：

1. 推理侧建议先保持当前基线，不再随意改动已经收敛的 `TP=2` 主链路。
2. 如果后续要支持更大模型或更长 prompt，优先继续扩充寒武纪 JSONL 表覆盖面。

4. 再决定是否补训练侧最小闭环。
5. 如果要满足任务书训练项，建议新增一条真正的训练 benchmark/estimator 链路，而不是继续把推理入口硬扩成训练。
